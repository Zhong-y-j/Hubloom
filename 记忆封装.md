## 记忆系统的设计目的

记忆系统的设计，是让「存什么、取什么、何时淘汰」从各个 Agent 里抽出来，集中到 `memory/` 统一管理。上层 Agent 只通过统一 API 读写记忆，专注智能体的推理与编排；下层由不同存储后端（SQLite、Qdrant、Neo4j）负责持久化与检索。

全项目用同一套 `remember` / `recall` 接口和分型返回（`RecallResult`），避免在每个 Agent 里重复写 SQLite、向量库、图数据库的调用代码。更换向量后端或图库时，主要改 `memory` 的 Handler / Store，不必动 Agent 核心逻辑。

`memory` 不负责：任务规划、工具编排、拼最终 prompt——这些由 Agent / `context` 完成。`memory` 只负责记忆本身的读写与生命周期。

---

## 四种记忆类型

四种类型 **逻辑用途不同**，**物理存储分离**。

| 维度         | Conversation（历史消息）     | Episodic（情节记忆）             | Semantic（语义记忆）             | Associative（联想图记忆）                            |
| ------------ | ---------------------------- | -------------------------------- | -------------------------------- | ---------------------------------------------------- |
| **记什么**   | 完整 `Message`               | 发生过的事实、事件摘要           | 稳定偏好、规则、抽象知识         | 实体 + 实体间关系                                    |
| **类比**     | 工作记忆 / 聊天窗口          | 情节记忆                         | 语义记忆                         | 联想记忆                                             |
| **隔离键**   | `session_id`                 | `namespace`                      | `namespace`                      | `namespace`                                          |
| **默认存储** | SQLite `conversation_memory` | Qdrant（`memory_type=episodic`） | Qdrant（`memory_type=semantic`） | Neo4j `:Entity` / `:MemoryRef`                       |
| **怎么取**   | 按时间最近 N 条              | query 向量 ANN                   | query 向量 ANN                   | 实体名 → 图邻域扩展                                  |
| **返回类型** | `list[Message]`              | `list[EpisodicItem]`             | `list[SemanticItem]`             | `AssociativeRecallResult`（在 `RecallResult.graph`） |
| **生命周期** | 仅 `clear_session`           | TTL + 容量驱逐                   | TTL + 容量（默认更长）           | 暂无自动驱逐                                         |

### 分工原则（不要混用）

- **对话原文** → 只进 **conversation**
- **可向量检索的短片段** → **episodic**（发生过的事）或 **semantic**（稳定知识/偏好）
- **实体关系** → **associative**

| 问题                                   | 用哪种记忆                               |
| -------------------------------------- | ---------------------------------------- |
| 「和这句话像不像？」                   | episodic / semantic（Qdrant 向量相似度） |
| 「A 和 B 什么关系、从 A 能联想到谁？」 | associative（Neo4j 图遍历）              |
| 「当前对话上下文是什么？」             | conversation                             |

**图与向量协作**：Neo4j 上用 `MemoryRef` 指向 Qdrant 的 `memory_id`；正文仍在向量库，图只存骨架与链接。

### 写入规范（避免污染）

```
每轮对话
  → conversation.remember(message=...)   # 必须带 role

会话结束 / LLM 提炼后
  → episodic.remember(content="事实摘要…")
  → semantic.remember(content="稳定偏好…")     # 勿把一次性事件写 semantic
  → associative.remember(metadata={from_name, to_name, relation_label, …})

禁止
  → 整段聊天记录直接写 episodic（应先提炼短句）
  → 长句/粗俗内容当实体名（实体用规范名，细节放 episodic + link_memory）
```

**namespace** 推荐格式 `{kind}:{owner}:{scope}`（如 `mem:tester_id:default`）。`create_memory_manager(namespace=...)` 时，该字符串同时作为 conversation 的 `session_id` 与长期记忆的隔离键。

## 多后端存储（默认生产路径）

| 后端                           | 承载的记忆          | 说明                                                               |
| ------------------------------ | ------------------- | ------------------------------------------------------------------ |
| **SQLite**（`data/memory.db`） | conversation        | 必选；`vector_backend="sqlite"` 时 episodic/semantic 也可回退到此  |
| **Qdrant**（单 Collection）    | episodic + semantic | 默认；用 payload 字段 `memory_type` 区分；共用 `QdrantMemoryStore` |
| **Neo4j**                      | associative         | 默认开启；`graph_backend="none"` 可不注册图记忆                    |

旧版 SQLite 内向量实现保留，工厂参数 `vector_backend="sqlite"` 可回退。

## 分层架构：Factory → Manager → Handler → Store

```
业务层（Hub / ReAct / API）
        │
        ▼
create_memory_manager()     memory/factory.py
        │  组装 Store、Embedder、各 Handler
        ▼
MemoryManager               memory/manager.py
        │  remember / recall / forget / clear_all / run_maintenance
        │
        ├─ ConversationHandler    → ConversationSQLiteStore    → SQLite
        ├─ EpisodicQdrantHandler  → QdrantMemoryStore (+ Embedder) → Qdrant
        ├─ SemanticQdrantHandler  → QdrantMemoryStore（共用）     → Qdrant
        └─ AssociativeHandler     → Neo4jStore                     → Neo4j
```

| 层级             | 职责                                                                       |
| ---------------- | -------------------------------------------------------------------------- |
| **Factory**      | 按 `vector_backend` / `graph_backend` 选择实现；统一 `namespace`           |
| **Manager**      | 对外唯一入口；参数校验（`message` vs `content` vs associative `metadata`） |
| **Handler**      | 单类记忆的业务语义：embed、拼 Item、生命周期、图三元组                     |
| **Store**        | 纯持久化与检索，不含中枢编排规则                                           |
| **lifecycle**    | TTL、容量驱逐（`QdrantMemoryStoreScope` 按 memory_type 限定）              |
| **embedders**    | 文本转向量（默认 `OpenAIEmbedder`，兼容 DashScope 等）                     |
| **consolidator** | 回合结束后用 LLM 提炼 episodic/semantic，并 `link_memory` 挂图             |

**原则**：Agent / Hub **只依赖 `MemoryManager` 或 factory**，不直接 `import` 具体 Store。

## `create_memory_manager` 工厂（`memory/factory.py`）

根据配置一次性组装可用的 `MemoryManager`：

```python
from memory.factory import create_memory_manager

mem = create_memory_manager(
    namespace="mem:tester_id:default",
    db_path="data/memory.db",
    vector_backend="qdrant",   # 或 "sqlite"
    graph_backend="neo4j",     # 或 "none"
    qdrant_collection="agentcortex_memory_v1024",
    embedder=None,             # 默认 OpenAIEmbedder()
)
```

| 参数                               | 作用                                                               |
| ---------------------------------- | ------------------------------------------------------------------ |
| `namespace`（必填）                | episodic/semantic/associative 隔离键；conversation 的 `session_id` |
| `db_path`                          | SQLite 路径（conversation 必选）                                   |
| `vector_backend`                   | `qdrant`（默认）或 `sqlite`（旧实现）                              |
| `graph_backend`                    | `neo4j`（默认）或 `none`（不启用联想记忆）                         |
| `qdrant_url` / `qdrant_collection` | 覆盖环境变量                                                       |
| `neo4j_uri`                        | 覆盖 Neo4j 连接                                                    |
| `embedder`                         | 可注入 Mock，便于测试                                              |

## `MemoryManager` 统一 API（`memory/manager.py`）

### 设计目标

1. **一个入口**：`remember` / `recall`，用 `memory_type` 区分四种记忆。
2. **返回分型**：`RecallResult` 按类型填充 `messages` / `items` / `graph`，避免混用。
3. **hybrid 有界**：默认 `hybrid` 只合并 **episodic + semantic**，不含 conversation 与 associative。

### `RecallResult`

```python
@dataclass
class RecallResult:
    memory_type: MemoryType | None
    messages: list[Message] | None = None           # conversation
    items: list[EpisodicItem | SemanticItem] | None # 向量长期记忆
    graph: AssociativeRecallResult | None = None    # associative
```

### `remember`（存储）

| memory_type             | 必填                       | 说明                                                       |
| ----------------------- | -------------------------- | ---------------------------------------------------------- |
| `conversation`          | `message: Message`         | 完整消息，含 role                                          |
| `episodic` / `semantic` | `content: str`             | 短句摘要；metadata 可含 `ref_session_id`、`importance`     |
| `associative`           | `content` 和/或 `metadata` | 如 `from_name`+`to_name` 建关系；`link_memory` 挂接 Qdrant |

### `recall`（检索）

| 调用                                    | 结果字段                                 |
| --------------------------------------- | ---------------------------------------- |
| `memory_type="conversation"`            | `.messages`（最近 N 条）                 |
| `memory_type="episodic"` / `"semantic"` | `.items`（向量 ANN + 分数阈值）          |
| `memory_type="associative"`             | `.graph`（实体邻域）                     |
| 不传 type，`mode="hybrid"`              | `.items`（episodic + semantic 去重合并） |

另有 `forget`、`clear_all`、`run_maintenance`（长期记忆的 TTL / 容量驱逐）。

## 与 ContextAssembler 的协作

**分工**：`memory` 负责取回结构化记忆；`context` 负责裁剪 Token、格式化成 prompt 块。

ReAct 每轮典型流程：

1. **`MemoryContextProvider.recall`**：`hybrid` 拉 episodic/semantic，可选 associative。
2. **`ContextAssembler.assemble`**：把记忆块与 conversation、RAG 等拼进 messages。

发给 LLM 的结构大致为：

```
[system: 人设 + 规则]
[system: [MEMORY]…[/MEMORY]]       ← hybrid 向量记忆摘要
[system: [GRAPH]…[/GRAPH]]         ← 图记忆摘要（可选）
[system: [DOCUMENTS]…[/DOCUMENTS]] ← RAG（retrieval 模块）
[conversation 最近 N 条 Message]
[user: 当前输入]
```

长期记忆与图记忆以 **文本摘要块** 注入，不破坏 tool 消息结构。

回合结束后：

- **conversation**：每轮写入用户句 + 助手句。
- **`MemoryConsolidator`**：用 LLM 从本轮对话提炼 → 写入 episodic/semantic/associative，并通过 **`link_memory`** 把图节点与 Qdrant 的 `memory_id` 关联。

---

## 一次完整调用（谁干什么）

```
用户发话
  → Agent：memory.recall（conversation + hybrid + 可选 associative）
  → MemoryContextProvider + ContextAssembler：拼 prompt
  → core LLM：生成回复
  → Agent：conversation.remember（本轮消息）
  → Consolidator（可选）：提炼长期记忆 + link_memory
```

**读**在调 LLM 之前；**写** conversation 几乎每轮都有；**写**长期记忆多为提炼后的短句，而非整段聊天原文。

---

## 一句话总结

**记忆封装：把四种用途不同的记忆、三种存储后端，收进 `MemoryManager` 统一接口；Agent 只管何时 recall/remember，ContextAssembler 只管怎么拼进 prompt；换存储、换向量实现主要改 memory 层，不动智能体核心逻辑。**
